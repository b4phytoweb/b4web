I. Informazioni Generali

L'API è basata su FastAPI e utilizza lo standard OAS 3.1. Versione API: 0.1.0.

La documentazione OpenAPI è disponibile via /openapi.json e la UI interattiva supporta l'operazione di Authorize per impostare token di accesso.

Nota: molte chiamate possono restituire 422 Validation Error in caso di payload non validi.

II. Gestione Utenti (Users Manager)

A. Accesso e Autenticazione

Endpoint principali per login, richiesta OTP e gestione password:

Endpoint Metodo Descrizione Body / Parametri
/users/request_access POST Richiede OTP per accesso (invia OTP a email dopo verifica credenziali). username, password (string)
/users/login POST Effettua login e restituisce token. username, password, otp
/users/change_password POST Cambio password (richiede vecchia password). new_password (string)
/users/reset_password POST Reset password tramite username + email. username, email, new_password

Tipicamente la risposta di /users/login ha la forma:

{
  "access_token": "eyJ... (jwt)",
  "token_type": "bearer",
  "user_type": "researcher"
}

B. Gestione Utenti (Amministratori)

Questi endpoint sono riservati ai superuser (admin).

  • /users/create_user POST — crea un utente: username, email, password, type.
  • /users/get_list GET — lista utenti con paginazione: page, limit. Restituisce total_users, total_pages, page, limit, users.
  • /users/delete/{user_id} DELETE — cancella utente (path param user_id).
  • /users/change_role PUT — cambia ruolo: body { "user_id": int, "new_role": "string" }.
  • /users/verify_email GET — verifica email per abilitare login: query param check.

III. Gestione Dati (Data Manager)

A. Gestione Metaboliti

Endpoint per CRUD sui metaboliti e per accedere ai dettagli e alle relazioni con specie e sperimentazioni.

  • /data/get_all_metabolites GET — lista completa di metaboliti.
  • /data/get_metabolite GET — dettaglio metabolita (query: metabolite_name).
  • /data/add_metabolite POST — aggiunta metabolita (payload complesso: name, cid, molecularformula, smile, InChIKey, dati spettrometria, tassonomia, ecc.).
  • /data/update_metabolite PUT — aggiornamento (modello metabolita).
  • /data/delete_metabolite/{name} DELETE — elimina per nome.
  • /data/metabolite_detail GET — dettagli estesi (query: metabolite_name).

B. Gestione Spettrometria

Endpoint per gestire dati sperimentali (mz, rt, adducts, peaks, immagini/file in base64).

  • /data/get_spectrometry GET — restituisce adduct e settings di base.
  • /data/add_spectrometry POST — inserisce dati spettrometria (parametri dettagliati, peaks, image_base64, file_base64).
  • /data/get_spectrometry/{id} GET — ottiene spettrometria per id.
  • /data/get_all_spectrometries GET — lista paginata.
  • /data/delete_spectrometry/{id} DELETE — rimozione per id; attenzione alle dipendenze.
  • /data/delete_all_metabolite_experiment DELETE — elimina tutti gli esperimenti di un metabolita (query: name).
  • /data/delete_metabolite_experiment DELETE — elimina esperimento specifico (query: name, rt, replicate_num, sample_id).

C. Gestione Tassonomia e Specie

  • /data/get_all_species GET — lista specie (paginata).
  • /data/get_specie GET — cerca specie e metaboliti associati (query: scientific_name).
  • /data/get_all_genera, /data/get_genus, /data/get_all_families, /data/get_family.
  • /data/add_taxonomy POST — aggiunge tassonomia (campi come id_taxa, scientific_name, kingdom, phylum, class_taxa, taxa_order, family, ...).
  • /data/update_taxonomy PUT — aggiorna tassonomia (by id).
  • /data/delete_taxonomy/{id_taxa} DELETE — elimina tassonomia (attenzione: cascata su altre tabelle).
  • /data/taxonomy_detail GET — dettaglio tassonomia con metaboliti e esperimenti (query: scientific_name).

D. Gestione Campioni e Località

  • /data/add_sample POST — aggiunge un campione (campi geo e tassonomici: istat_code, date, taxonomy_id, locality, decimalLatitude, occurrenceRemarks).
  • /data/get_all_samples GET — lista campioni con paginazione.
  • /data/get_sample/{sample_id} GET — dettaglio campione.
  • /data/update_sample/{sample_id} PUT — aggiornamento campione.
  • /data/delete_sample/{sample_id} DELETE — eliminazione (attenzione alle dipendenze).
  • Varie utility: /data/search_occurrence, /data/get_all_sample_localities, /data/get_all_sample_identifies.
  • Endpoint geografici: /data/get_comuni, /data/get_province, /data/get_all_provinces, /data/get_sigla.

E. Ricerca Dati (Metaboliti e Spettrometria)

Funzionalità di ricerca avanzata basate su m/z sperimentale, formula + RT, o frammenti:

  • /data/search_metabolite_by_experimental_mz GET — query: experimental_mz, experimental_mz_type (1/-1), error_window_mz.
  • /data/get_metabolite_by_neg_mz_expected GET — ottiene Mz negative attese (param: name).
  • /data/search_metabolite_by_formula_rt GET — ricerca per molecular_formula e rt (+ finestre di errore).
  • /data/search_metabolite_by_fragments GET — ricerca per frammenti (array di frammenti, tipo, finestra di errore).

F. Gestione Settings e Adducts

  • /data/get_all_adducts GET — lista adduct distinti per tipo (adduct_type=pos|neg).
  • /data/get_all_settings GET — lista settings.
  • /data/get_setting GET — ricerca setting per parametri (ionization_source, ionization_type, device_model, collision_energy, capillary_voltage, conical_voltage).
  • /data/add_setting POST, /data/update_setting/{id} PUT, /data/delete_setting/{id} DELETE.

G. Gestione Gruppi di Ricerca e Composti

  • /data/get_all_compounds GET — composti (paginati).
  • /data/get_all_research_groups GET — gruppi di ricerca (paginati).

H. Gestione Replicati Biologici

  • /data/add_biological_replicate POST — aggiunge replicato (campi: number, sample_id, setting_id?, notes?, scientific_name).
  • /data/delete_biological_replicate DELETE — elimina replicato (query: number, sample_id).
  • /data/get_list_biologicalreplicate GET — lista replicati (filtri: sample_id, paginazione).

I. Gestione Bioattività e Classi

Endpoint per bioassay, bioactivities e classi:

  • /data/get_bioassays_from_db GET — bioassay dal DB per un metabolita (query metabolite_name).
  • /data/get_all_bioactivities GET — lista bioactivities (paginata).
  • /data/get_bioactivities_for_taxonomy/{scientific_name} GET — bioactivities per tassonomia.
  • /data/get_bioactivity/{bioactivity_id}, /data/add_bioactivity POST, /data/update_bioactivity PUT, /data/delete_bioactivity/{id} DELETE.
  • Endpoint per gestione classi: /data/get_all_bioactivity_classes, /data/add_bioactivity_class, /data/update_bioactivity_class/{id}, /data/delete_bioactivity_class/{id}.
  • Analoghi endpoint per bioassay classes.

J. Ricerca per Sinonimi

  • /data/metabolite_synonyms GET — ricerca sinonimi metaboliti (query: synonym).
  • /data/taxonomy_synonyms GET — ricerca tassonomia per sinonimi (query: synonym).

IV. Machine Learning

Endpoint dedicato al confronto / matching spettri:

  • /ml/spectrum_match POST — form multipart con file JSON binario contenente spettri (input_file), opzionali machine (default QTOF), ion_mode (POSITIVE o NEGATIVE), compound_class (default Natural Product), opzionale top_n.

Tipicamente la risposta è un oggetto con le top N corrispondenze e punteggi di similarità; payload e parametri sono multipart/form-data.

V. Connettore PubChem (Pubchem connector)

Call che interrogano PubChem per arricchire i dati interni:

  • /conn/pubchem/get_metabolite_data GET — query: name restituisce informazioni pubbliche (es. CID, proprietà chimiche).
  • /conn/pubchem/get_pubchem_taxonomy GET — dato il nome di un esemplare restituisce tassonomia.
  • /conn/pubchem/get_pug_view_data GET — dato il cid restituisce la pagina PUG-View completa.
  • /conn/pubchem/get_bioassays_from_cid GET — bioassay da PubChem per CID.
  • /conn/pubchem/get_bioassays_from_aid GET — bioassay da PubChem per AID (paginato con page, limit).

VI. Connettore Excel (Excel connector)

Endpoint per import massivo da file Excel/CSV e per generare grafici di spettrometria dal file:

  • /conn/excel/spectrometry_chart POST — carica file (.xlsx/.xls) e parametro intensity_threshold per generare grafico dei picchi.
  • /conn/excel/add_bioactivity POST — importa bioactivity da file xlsx/csv.
  • /conn/excel/add_taxonomy_and_sample POST — importa tassonomia e/o campioni.
  • /conn/excel/add_experimental_metabolite POST — importa experimental metabolite da CSV/XLSX/XLS.

Questi endpoint usano multipart/form-data per l'upload dei file.

VII. Schemi dei Dati (Schemas)

Riepilogo degli schemi principali (modelli Pydantic / OpenAPI):

Schemi Utente

  • LoginRequest: { username: string, password: string, otp?: string }
  • LoginResponse: { access_token: string, token_type: string, user_type: string }
  • CreateUserRequest: { username: string, email: string, password: string }
  • ChangeUserRoleRequest: { user_id: integer, new_role: string }

Metaboliti e Tassonomia

  • MetaboliteModel: modello complesso con campi quali name, molecularformula, InChIKey, smile, molecularclass, neg_mz, pos_adduct, ...
  • MetaboliteDetailResponse: { data: object, classes: array, unique_neg_fragments: array, unique_pos_fragments: array }
  • TaxonomyModel: { id_taxa: string, scientific_name: string, kingdom: string, phylum: string, class_taxa: string, taxa_order: string, family: string }
  • TaxonomyDetailResponse: include dettagli tassonomici, metaboliti e campioni associati.

Schemi Campioni (Samples)

  • SampleModel: include istat_code, date, taxonomy_id, locality, decimalLatitude, occurrenceRemarks, notes e altri campi di contesto.
  • OccurrenceRemarksItem: { sample_id, scientific_name, municipality, province_abbreviation, replicate_number }

Schemi Spettrometria e Settings

  • SpectrometryRequestModel: { metabolite_name, rt: number, intensity_threshold: number, ionization: 1|-1, adduct: string, device: {...}, filtered_peaks: [{mz, intensity, percentBIC}], image_base64?, file_base64? }
  • SettingModel: { ionization_source, ionization_type, device_model, collision_energy: number, capillary_voltage: number, conical_voltage: number }

Schemi Bioattività

  • BioactivityInsertRequest: struttura dettagliata per collegare bioactivity a campioni, tassonomia e settings.
  • BioactivityClassRequest / Response: { id: string, bioactivity: string }
  • BioassayClassRequest / Response: { id: string, bioassay: string }

Nota sull'uso degli schemi

Per implementare client robusti si consiglia di scaricare /openapi.json e generare client (es. openapi-generator o swagger-codegen) in modo da avere tipi e validazioni coerenti lato client.

Analogia operativa

Per semplificare la comprensione: il sistema API funziona come una biblioteca specializzata:

  • Users Manager = ufficio di registrazione (tessere / ruoli).
  • Data Manager = sale di lettura e cataloghi (metaboliti, spettrometria, tassonomia, campioni).
  • Connettori = servizi esterni o import (PubChem, Excel).
  • Ricerca = motore di ricerca avanzato (per formula, frammenti, m/z sperimentale).
